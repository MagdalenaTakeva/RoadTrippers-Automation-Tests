# RoadTrippers Automation Tests

This project contains automated UI tests for the RoadTrippers web application using Selenium and pytest. The focus is on testing features like adding stops to a trip, with a structure that supports easy expansion to other pages (e.g., login, dashboard, map interactions).

## Project Structure

The project follows a modular structure optimized for Selenium-based UI automation testing with pytest. It uses the Page Object Model (POM) for maintainability, fixtures for setup/teardown, factories for test data generation, and CircleCI for CI/CD. Key principles include:

- **Separation of concerns**: Tests are isolated from page logic (in `pages/`), fixtures handle shared setup (e.g., drivers), and utilities like factories provide consistent data.
- **Test isolation**: Function-scoped fixtures ensure fresh browsers per test, preventing session carry-over.
- **CI-friendly**: Parallelism for stateless tests, sequential for authenticated ones to avoid race conditions.
- **Data-driven**: Factories generate deterministic location data for reliability.
- **Screenshots & Logs**: Automatic captures on failure for debugging.

- **Root Files**: Core config (conftest.py, pytest.ini) for pytest discovery/marking/logging. Requirements for deps. Save_session.py for cookie export.
- **Hidden/Generated**: .venv/ (virtual env), reports/ (HTML/JUnit from CI), .gitignore (ignores .pkl, venv, screenshots, etc.).
- **No src/ Folder**: Pages/fixtures/factories are top-level for simplicity; no production code—pure tests.

This structure ensures scalability: Add new pages/tests easily, maintain isolation, and debug via logs/screenshots.

## Setup Instructions

1. **Clone the repository**  
   ```
   git clone <repo-url>
   cd RoadTrippersProject
   ```

2. **Create and activate virtual environment**  
   On Windows:  
   ```
   python -m venv .venv
   .venv\Scripts\activate.bat
   ```  
   On macOS/Linux:  
   ```
   python -m venv .venv
   source .venv/bin/activate
   ```

3. **Install dependencies**  
   ```
   pip install -r requirements.txt
   ```  
   (See [requirements.txt](#dependencies) below for the list.)

4. **Configure PyCharm (if using)**  
   - Open the project in PyCharm.  
   - Set Python Interpreter: Project → Python Interpreter → Add existing → `.venv\Scripts\python.exe` (Windows) or `.venv/bin/python`.  
   - Mark directories: Right-click `tests/` → Mark Directory as → Test Sources Root (green).  
   - (Optional) Mark `pages/` or `fixtures/` as Sources Root (blue).  
   - In Settings → Tools → Python Integrated Tools: Set Default test runner to pytest.

## Handling Authentication: Bypassing Bot Detection with Manual Login and Cookie Persistence

RoadTrippers employs anti-bot measures to protect against automated access, particularly during login. These measures can detect Selenium/WebDriver usage through fingerprints like `navigator.webdriver=true`, headless mode indicators, unnatural interaction patterns, or server-side behavior analysis. As a result:

- Manual browser login succeeds with valid credentials.
- Automated Selenium login often fails with a generic "invalid username or password" error, even when credentials are correct. This is not due to bad credentials but upstream rejection of the automated request.

Attempting to evade detection (e.g., via stealth plugins, fingerprint spoofing, or proxies) is not recommended for this project, as it risks violating terms of service and complicates maintenance. Instead, use an ethical, professional workaround: **perform a one-time manual login, save the authenticated session cookies, and reuse them in tests to skip the login form entirely**.

This approach:
- Avoids triggering bot detection by bypassing form submission.
- Simulates a returning authenticated user.
- Is stable for short-lived sessions (cookies typically expire in days/weeks).
- Is suitable for testing/demo purposes (use test accounts to avoid impacting real data).

### Step-by-Step Implementation

1. **Export Cookies Script (`save_session.py`)**  
   Run this once to manually log in and save cookies:

   ```python
   import pickle
   from selenium import webdriver
   from selenium.webdriver.chrome.service import Service
   from selenium.webdriver.chrome.options import Options
   from selenium.webdriver.common.by import By
   from selenium.webdriver.support.ui import WebDriverWait
   from selenium.webdriver.support import expected_conditions as EC

   options = Options()
   driver = webdriver.Chrome(service=Service(), options=options)

   driver.get("https://roadtrippers.com")

   print("👉 Please log in manually in the opened browser.")
   print("👉 The script will automatically detect when login succeeds.")

   # Wait until avatar (logged-in indicator) appears
   avatar_locator = (By.CSS_SELECTOR, ".rt-user-img [href*='/people']")

   WebDriverWait(driver, 300).until(
       EC.presence_of_element_located(avatar_locator)
   )

   print("✅ Login detected. Saving cookies...")

   pickle.dump(driver.get_cookies(), open("roadtrippers_cookies.pkl", "wb"))

   print("✅ Cookies saved successfully.")
   driver.quit()
   ```

   - Run: `python save_session.py`  
   - This saves `roadtrippers_cookies.pkl` in the project root (add to `.gitignore`).

2**Troubleshooting Cookie Issues**  
   - **Expiration**: If tests fail with unauthenticated errors, re-run the export script (cookies expire).  
   - **Domain Mismatch**: Always `get()` the base domain before adding cookies.  
   - **Invalid Cookies**: Some may be HttpOnly/Secure — Selenium can't access all, but auth tokens usually work.  
   - **Multiple Environments**: Save separate `.pkl` files for different accounts/environments.  
   - **Headless Mode**: If using `options.add_argument('--headless')`, detection might still occur — test with headed first.

This method ensures reliable, detection-free access while adhering to ethical automation practices.

## Running Tests

- **From Terminal** (with venv active):  
  ```
  pytest tests/ -v  # Run all tests verbosely
  pytest tests/add_stops_page_tests.py --html=report.html  # With HTML report
  pytest --cov=pages/ --cov-report=html  # Coverage report (adjust to pages/ or full project)
  ```

- **From PyCharm**: Right-click test file/function → Run/Debug 'pytest in ...'

- **Parallel Runs** (speedup for large suites):  
  ```
  pytest -n auto
  ```

- **Flaky Tests** (rerun failures):  
  ```
  pytest --reruns 2 --reruns-delay 5
  ```

- **By Marker**:  
  ```
  pytest -m smoke     # Quick smoke tests
  pytest -m auth      # Authenticated tests (sequential in CI)
  pytest -m negative  # Negative/edge cases
  ```

## Reporting

- HTML reports: Generated via `--html=report.html` (open in browser).  
- Coverage: Open `htmlcov/index.html`.  
- Allure (if enabled): `pytest --alluredir=allure-results`, then `allure serve allure-results`.
